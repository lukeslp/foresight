1. **GET /api/current returning 404 when no prediction cycles exist**:
   - **Is 200 with {cycle: null, status: 'no_cycles'} better than 404?**
     Yes, returning a 200 status code with a payload like `{cycle: null, status: 'no_cycles'}` is often better than a 404 in this context. A 404 status code indicates that the resource could not be found, which might imply to the client that the endpoint itself doesn't exist or is incorrect. In your case, the endpoint exists, but there is simply no data to return yet (an empty state). A 200 status with a clear indication of the empty state communicates that the request was successful, but there is no active cycle. This approach avoids unnecessary error handling on the frontend for a situation that isn't truly an error.
   - **What about 204?**
     A 204 status code means "No Content" and is typically used when the server has successfully processed the request but has no content to return. While it could work, it might be less informative for the frontend since it doesn't provide any context about why there is no content (e.g., whether it's an empty state or some other reason). If you use 204, the frontend would need to infer the reason for no content, which could lead to ambiguity. Therefore, 200 with a descriptive payload is generally preferable.
   - **Other considerations for 'empty state' vs 'not found' semantics in REST?**
     In REST design, distinguishing between "empty state" and "not found" is important for clarity. A 404 should be reserved for situations where the requested resource or endpoint does not exist (e.g., a specific cycle ID that never existed). For an empty state (e.g., no cycles exist yet), a 200 with an empty or null result, or even a 200 with metadata explaining the situation, is more appropriate. This helps the client differentiate between a genuine error (404) and a valid but empty result. Additionally, consider documenting these behaviors in your API specification (e.g., OpenAPI/Swagger) so that frontend developers know what to expect.

2. **Is 503 appropriate for a /health endpoint when a background worker thread isn't running?**
   - Yes, a 503 "Service Unavailable" status code can be appropriate for a `/health` endpoint if a critical component like a background worker thread is not running, assuming that the absence of this worker means the service cannot function properly. A 503 indicates that the server is temporarily unable to handle the request due to a temporary condition (like a missing worker). However, consider the following:
     - If the background worker's absence doesn't completely impair the service (e.g., the API can still respond to some requests), you might instead return a 200 with a payload indicating partial health (e.g., `{status: 'degraded', worker: 'not_running'}`).
     - Reserve 503 for situations where the service is truly unavailable or cannot fulfill its primary purpose without the worker.
     - Document the behavior of the `/health` endpoint clearly so that monitoring tools or clients can interpret the status correctly.
